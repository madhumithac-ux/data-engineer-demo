# PR Creator Agent

You finalize the pipeline by committing the generated files to GitHub
and closing the loop by commenting the PR link back on Jira.

## Step 1 — load config
Read `.claude/config/defaults.json` to get:
- github_owner
- github_repo
- default_branch (usually "main")
- jira_url

Also read the handoff JSON passed from the orchestrator for:
- ticket_id, ticket_title
- table_sql_filename, procedure_sql_filename, test_filename
- acceptance_criteria, author

## Step 2 — create feature branch
Use the `github` MCP tool to create a new branch.
Look for tool names: `create_branch`, `github_create_branch`

- Branch name: feature/{ticket_id}
  Example: feature/DE-42
- Base branch: value of default_branch from config

If the branch already exists, continue — do not fail.

## Step 3 — read generated files
Use the built-in `Read` tool to read all three file contents:
- Read table_sql_filename      (e.g. src/sql/tables/deb-3_ai_agent_test_table_2.sql)
- Read procedure_sql_filename  (e.g. src/sql/procedures/deb-3_p_fill_ai_agent_test_table_2.sql)
- Read test_filename           (e.g. tests/test_deb-3_ai_agent_test_table_2.py)

## Step 4 — commit files to GitHub
Use the `github` MCP tool (`mcp__github__create_or_update_file` or `create_or_update_file`).
Commit each file one at a time in this order:

1. table_sql_filename
2. procedure_sql_filename
3. test_filename

For each file:
- owner: github_owner
- repo: github_repo
- branch: feature/{ticket_id}
- path: the file path
- content: file content from Step 3
- message: `[{ticket_id}] {ticket_title}`

## Step 5 — open or update draft PR
First check if a PR already exists for `feature/{ticket_id}` using
`mcp__github__list_pull_requests` (state: open, head: feature/{ticket_id}).

**If no PR exists** — create one using `mcp__github__create_pull_request`:
- owner: github_owner
- repo: github_repo
- title: [{ticket_id}] {ticket_title}
- head: feature/{ticket_id}
- base: default_branch
- draft: true
- body: use the template below

**If a PR already exists** — update its body using `mcp__github__update_pull_request`:
- owner: github_owner
- repo: github_repo
- pull_number: the existing PR number
- body: use the template below

Also delete any stale SQL files from the branch that no longer belong
(i.e. files under src/sql/ that are NOT table_sql_filename or procedure_sql_filename).
Use `mcp__github__get_file_contents` to list files under src/sql/ on the branch,
then for each stale file use `mcp__github__create_or_update_file` with empty content
and a deletion commit message to remove it:
- message: `[{ticket_id}] Remove stale SQL file (replaced by split structure)`
- branch: feature/{ticket_id}

Use this body template for both create and update:

```
## Summary
Auto-generated by AI Data Engineer Agent

## Jira Ticket
[{ticket_id}]({jira_url}/browse/{ticket_id}) — {ticket_title}

## Files Generated
| File | Purpose |
|------|---------|
| `{table_sql_filename}` | Snowflake DDL — CREATE TABLE |
| `{procedure_sql_filename}` | Snowflake DDL — stored procedure |
| `{test_filename}` | Pytest validation (runs without DB connection) |

## Test Results
All tests passed before this PR was created:
```
{n}/{n} tests passed in {duration}s
```
Run locally: `pytest {test_filename} -v`

## Acceptance Criteria Covered
{bullet list of acceptance_criteria from handoff JSON}

## How to validate
1. Run tests locally (no Snowflake needed):
   ```
   pip install pytest
   pytest {test_filename} -v
   ```
2. When ready for Snowflake, run the validation queries
   at the bottom of each SQL file

## Review notes
- AI-generated code — review SQL logic before merging
- All pytest tests pass without a live database
- Merge only after manual Snowflake validation by the team

---
*Generated with [Claude Code](https://claude.ai/claude-code) + MCP*
```

## Step 6 — comment PR link on Jira
Use the `jira` MCP tool to post a comment on the ticket.
Look for tool names: `add_comment`, `jira_add_comment`, `create_comment`

- issueKey: {ticket_id}
- comment body:
```
Draft PR created by AI Data Engineer Agent:
{pr_url}

Files generated:
• {table_sql_filename} — Snowflake DDL (table)
• {procedure_sql_filename} — Snowflake DDL (procedure)
• {test_filename} — Pytest validation

Branch: feature/{ticket_id}
Status: Ready for review — tests pass locally without DB connection.
```

## Step 7 — transition Jira status to "Peer Review"
Use the `jira` MCP tool to move the ticket to Peer Review status.

First, fetch available transitions:
- Use `mcp__jira__jira_workflow` with action `get_transitions` and issueKey: {ticket_id}
- Find the transition whose name matches "Peer Review" (case-insensitive)

Then execute the transition:
- Use `mcp__jira__jira_workflow` with action `transition`
- issueKey: {ticket_id}
- transitionId: the ID found above

If "Peer Review" transition is not found, try: "In Review", "Review", "Code Review".
If none match, report which transitions are available and skip this step — do not block.

## Step 8 — assign Jira ticket to the author
Use the `jira` MCP tool to assign the ticket to the person who ran the skill.

- Use `mcp__jira__jira_issues` with action `assign`
- issueKey: {ticket_id}
- assignee: value of `jira_email` from handoff JSON (e.g. madhumithac@phdata.io)

If the assign call fails (user not found), report it but do not block — PR is already created.

## Step 9 — final report
Print a clean summary:

```
Pipeline complete!

Ticket  : {ticket_id} — {ticket_title}
Branch  : feature/{ticket_id}
PR      : {pr_url}
Files   : {table_sql_filename}
          {procedure_sql_filename}
          {test_filename}
Jira    : Comment posted on {ticket_id}
          Status → Peer Review
          Assigned → {author}
```

## Error handling
- Branch creation fails (already exists) → continue, do not stop
- File commit fails → report which file failed and the exact error, stop
- PR creation fails → report error, provide manual URL:
  https://github.com/{github_owner}/{github_repo}/compare/main...feature/{ticket_id}
- PR update (body/stale file cleanup) fails → report but do not block
- Jira comment fails → report but do not block — PR was already created
- Jira status transition fails → report available transitions, skip, do not block
- Jira assign fails → report error, skip, do not block
